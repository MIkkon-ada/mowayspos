# UAT 测试计划

本文档用于上线前 UAT。目标是让真实业务角色在接近生产的环境中完成关键流程确认。

## 1. UAT 目标

- 验证同域部署下的登录、路由、权限、提交、确认、决策和退出流程
- 验证 Dashboard 首页风格保持冻结
- 验证异常页面和刷新行为没有 404 / 白屏
- 验证上线前备份 / 恢复链路可用

## 2. 测试角色

- 公司 CEO
- 企业教练
- 项目负责人 / PM
- 项目统筹人
- 关键任务责任人
- 关键任务协助人

## 3. 测试项目准备

- 准备至少 1 个可用测试项目
- 准备一个已知成员关系的项目
- 准备一个用于角色切换的项目
- 如有必要，使用已归档的 TEST_* 项目作为参考样例，但不要影响正式统计

## 4. 每个角色的操作

### 公司 CEO

- 登录
- 进入管理页
- 查看项目列表
- 查看成员管理
- 查看 AI 确认中心
- 验证可访问全局管理入口

### 项目负责人 / PM

- 登录
- 进入项目首页
- 创建或查看进展提交
- 打开 AI 确认中心
- 确认提交
- 查看 Dashboard

### 关键任务协助人

- 登录
- 进入项目首页
- 提交进展
- 查看自己可见的内容
- 验证无法访问管理入口

### 项目统筹人

- 登录
- 查看转交给协调人的待办
- 完成协调反馈
- 验证无法执行越权确认

### 企业教练

- 登录
- 查看企业教练决策待办
- 执行企业教练决策
- 验证无法执行越权写入

## 5. 通过 / 失败判定

通过标准：

- 页面无白屏
- 深链刷新不 404
- `GET /api/health` 正常
- 登录 / 退出流程正常
- 权限边界符合角色定义
- 关键提交流程可闭环

失败标准：

- 页面白屏
- 任一路由刷新 404
- 登录失败或退出后仍保持登录态
- 角色越权
- API 反代异常
- Cookie 属性不符合要求

## 6. 问题记录模板

记录项：

- 日期
- 角色
- 页面 / 路由
- 操作步骤
- 预期结果
- 实际结果
- 问题等级
- 截图或日志
- 是否复现
- 是否阻塞上线

## 7. UAT 结束后是否允许上线的判定

允许上线的前提：

- 所有关键角色流程通过
- 真实服务器验证通过
- 备份 / 恢复 SOP 可执行
- 无 P0 阻塞项
- 已确认数据库与部署配置可回滚

不允许上线的情形：

- 任一关键角色流程失败
- 任一 P0 阻塞未解除
- 真实服务器验证未完成
- 备份 / 恢复不可用

## 8. 备注

- UAT 不是开发测试替代品
- UAT 需要尽量接近真实部署环境
- 若无真实 Linux 服务器，应先完成环境准备，再进入 UAT
