# Moways-SOP 主流程阶段基准表

> 本文档是 N4-P0-BASELINE-MAIN-FLOW-STAGE-MAP 的流程基准产物。  
> 目标是先固定“从项目创建到归档”的主流程阶段边界，作为后续 Codex 修改指令的判断依据。本文只基于当前代码事实与已确认业务口径梳理，不修改业务代码。

## 1. 总体结论

当前系统的主流程已经具备一条可运行的 V1.0 骨架：

项目创建 → 立项下发 → 负责人完善立项信息与工作推进表雏形 → 企业教练审核 → 项目进入 active → 工作推进表执行 → 成员提交工作进展 → AI 提取 → 负责人确认入库 → 成果 / 问题 / 工作推进表 / 驾驶舱回流 → 归档。

但当前最大风险不是单个按钮或页面，而是“主流程阶段边界不够清晰”。特别需要固定以下口径：

- Project = 项目。
- Task = 重点工作。
- SubTask = 关键任务。
- 项目成员不等于关键任务责任人。
- 项目 active 不等于每个成员都已有可汇报任务。
- AI 负责提取内容，不负责决定业务归属。
- AI 提取结果不是正式数据；负责人确认后才写入正式数据。
- AI 确认中心应是“待确认处理台”，不是“我的提交记录中心”。
- 工作推进表雏形不是最终执行版；V1.0 当前以“企业教练审核通过”作为雏形确认成执行版的节点。

当前代码已存在状态：

| 状态 | 业务含义 |
|---|---|
| `draft` | 项目草稿，尚未下发。 |
| `dispatched` | 项目已下发给负责人，等待负责人完善立项信息和工作推进表雏形。 |
| `pending_review` | 负责人已提交立项信息和工作推进表雏形，等待企业教练审核。 |
| `returned` | 企业教练退回，负责人需要修改后重新提交。 |
| `active` | 项目进入执行阶段。V1.0 暂理解为工作推进表雏形已被确认为执行版。 |
| `archived` | 项目已归档，只能查看，不应继续提交汇报或新增任务。 |

## 2. 主流程总览图

```text
项目创建
  ↓
立项下发
  ↓
负责人完善立项信息
  ↓
形成工作推进表雏形
  ↓
企业教练审核
  ↓
启动确认 / 启动会
  ↓
形成正式执行版工作推进表
  ↓
关键任务派发
  ↓
成员按关键任务提交进展
  ↓
AI 提取
  ↓
负责人确认入库
  ↓
成果库 / 问题中心 / 工作推进表 / 驾驶舱回流
  ↓
项目结束 / 归档
```

V1.0 当前实现说明：

- 当前没有独立的 `pending_kickoff` / `kickoff_review` 状态。
- 当前企业教练审核通过后，`approve_project()` 直接将项目置为 `active`，并写入 `kickoff_date / kickoff_by`。
- 因此 V1.0 口径必须明确：企业教练审核通过 = 当前工作推进表雏形确认为执行版，项目进入 active。
- 后续增强版可以增加“待启动 / 启动确认”状态，但不能在未设计前用会议纪要或页面文案替代状态边界。

## 3. 阶段 1：项目创建

1. 阶段名称：项目创建。
2. 通俗理解：管理员或有权限的人先建立一个项目壳，配置基础资料和角色。
3. 主要角色：super_admin / tech_admin；部分管理入口可能允许 company_ceo 查看，但不应自动成为项目企业教练。
4. 入口页面：`/home/projects`，前端组件 `ProjectsMgmtSection`。
5. 核心操作：创建项目、填写项目名称/类型/客户/周期、配置企业教练/负责人/统筹人/成员。
6. 核心产物：Project 记录、ProjectMember 角色关系。
7. 数据对象：`models.Project`、`models.ProjectMember`。
8. 状态变化：新建项目应默认为 `draft` / `is_active=False`。
9. 下一阶段入口：项目管理页“下发项目”。
10. 该阶段允许做什么：创建草稿、编辑项目基础资料、配置角色。
11. 该阶段禁止做什么：让项目直接进入执行；让普通成员提交正式工作汇报；创建第四层任务。
12. 当前代码是否已经支持：基本支持。`models.Project.status` 默认 `draft`，`ProjectCreatePayload.status` 默认 `draft`。
13. 当前代码差异 / 风险：`excel_importer.py`、`permissions.py` 中仍有旧导入/种子路径直接写 `is_active=True` 的痕迹，属于历史兼容入口，应后续审计是否仍被正式流程使用。

## 4. 阶段 2：立项下发

1. 阶段名称：立项下发。
2. 通俗理解：项目已准备好基础角色，正式下发给项目负责人完善立项资料。
3. 主要角色：super_admin / tech_admin / 有项目管理权限的人。
4. 入口页面：`ProjectsMgmtSection` 项目队列中的“下发项目”。
5. 核心操作：调用 `dispatchProject()` → 后端 `POST /api/projects/{project_id}/dispatch`。
6. 核心产物：下发通知、项目进入待负责人完善阶段。
7. 数据对象：Project、Notification、OperationLog。
8. 状态变化：`draft` → `dispatched`。
9. 下一阶段入口：负责人打开 OwnerSubmitModal 完善立项信息。
10. 该阶段允许做什么：下发给负责人、通知负责人。
11. 该阶段禁止做什么：直接生成正式执行工作推进表；让成员正式汇报。
12. 当前代码是否已经支持：支持。`dispatch_project()` 会设置 `dispatched`。
13. 当前代码差异 / 风险：下发前角色完备性主要由前端提示和后端校验共同保证，仍需保持 owner / project_ceo 边界，不允许 company_ceo 自动充当项目企业教练。

## 5. 阶段 3：负责人完善立项信息

1. 阶段名称：负责人完善立项信息。
2. 通俗理解：PM / 项目负责人补全项目目标、周期、完成准则和补充信息。
3. 主要角色：项目 owner；super_admin / tech_admin 可兜底。
4. 入口页面：OwnerSubmitModal（当前为弹窗工作台，不是独立页面）。
5. 核心操作：填写项目名称、项目周期 / 时间段、项目完成准则 / 验收标准、补充详细信息。
6. 核心产物：可提交审核的项目立项资料。
7. 数据对象：Project、`ProjectProfilePayload`。
8. 状态变化：本阶段编辑过程中仍是 `dispatched` 或 `returned`；提交后进入 `pending_review`。
9. 下一阶段入口：同一弹窗内维护工作推进表雏形并提交审核。
10. 该阶段允许做什么：完善项目核心资料、补充客户/背景/说明。
11. 该阶段禁止做什么：提交正式工作汇报；确认入库；修改数据库结构；新增 workstream 表。
12. 当前代码是否已经支持：支持。`ownerSubmitProfile()` 调用 `POST /api/projects/{project_id}/owner-submit`。
13. 当前代码差异 / 风险：OwnerSubmitModal 的业务口径已收口，但它仍承担“立项填写 + 工作推进表雏形编制”两类职责，后续修改必须避免把它扩展成执行期工作推进表。

## 6. 阶段 4：工作推进表雏形形成

1. 阶段名称：工作推进表雏形形成。
2. 通俗理解：负责人在立项阶段先编制 Task / SubTask 草案，作为后续执行版的基础。
3. 主要角色：项目 owner。
4. 入口页面：OwnerSubmitModal 右侧“工作推进方案”。
5. 核心操作：维护重点工作、目标成果 / 验收标准、关键任务、责任人、协助人、时间段、备注 / 标准。
6. 核心产物：Task 草案、SubTask 草案。
7. 数据对象：复用 `models.Task` 和 `models.SubTask`，不新增表。
8. 状态变化：提交后项目进入 `pending_review`；Task/SubTask 作为工作推进表基础数据保留。
9. 下一阶段入口：企业教练审核材料。
10. 该阶段允许做什么：创建重点工作与关键任务草案。
11. 该阶段禁止做什么：新增第四层任务；新增 Workstream 表；把 Task / SubTask 语义改乱。
12. 当前代码是否已经支持：支持。`_save_work_progress_draft()` 会把 `work_progress_draft` 写入 Task/SubTask。
13. 当前代码差异 / 风险：雏形和执行版当前共用 Task/SubTask 表，没有额外版本状态。后续如需要“草案版/执行版”差异，需要设计而不是临时加字段。

## 7. 阶段 5：企业教练审核

1. 阶段名称：企业教练审核。
2. 通俗理解：项目企业教练审核项目完成准则、重点工作和关键任务安排。
3. 主要角色：项目层 `project_ceo`；super_admin / tech_admin 兜底。系统层 `company_ceo` 不能自动成为项目企业教练。
4. 入口页面：ProjectsMgmtSection 的“查看审核材料 / 审核立项”。
5. 核心操作：审核通过或退回修改。
6. 核心产物：审核意见、项目状态流转。
7. 数据对象：Project、ProjectMember、Task、SubTask、Notification、OperationLog。
8. 状态变化：`pending_review` → `active` 或 `returned`。
9. 下一阶段入口：审核通过后进入工作推进表；退回后回到负责人修改。
10. 该阶段允许做什么：只读查看立项方案、通过、退回。
11. 该阶段禁止做什么：让 company_ceo 自动处理项目教练审核；在审核材料视图中编辑负责人提交内容。
12. 当前代码是否已经支持：支持。`approve_project()`、`return_project()` 使用项目教练 / 管理员权限。
13. 当前代码差异 / 风险：审核通过直接 `active`，没有独立启动确认阶段；这是 V1.0 明确口径，不是遗漏修复。

## 8. 阶段 6：启动确认 / 启动会

1. 阶段名称：启动确认 / 启动会。
2. 通俗理解：理想流程中，企业教练审核后的工作推进表雏形应在启动会上被讨论并确认成执行版。
3. 主要角色：企业教练、负责人、统筹人、成员。
4. 入口页面：当前没有独立启动确认页面；会议纪要模块支持 `meeting_type=kickoff`。
5. 核心操作：理想状态是确认雏形成为执行版；当前 V1.0 由企业教练审核通过隐含完成。
6. 核心产物：正式执行版工作推进表、启动会纪要（可选）。
7. 数据对象：Project、Task、SubTask、Meeting。
8. 状态变化：理想增强版可设计 `pending_kickoff` → `active`；当前 V1.0 为 `pending_review` → `active`。
9. 下一阶段入口：`/work/tasks` 工作推进表。
10. 该阶段允许做什么：沉淀会议纪要、讨论任务安排。
11. 该阶段禁止做什么：用会议纪要类型替代正式生命周期状态；未经设计新增状态。
12. 当前代码是否已经支持：部分支持。会议纪要支持 kickoff 类型；项目审核通过会写 `kickoff_date/kickoff_by`。
13. 当前代码差异 / 风险：没有强制启动会状态。P1 增强项，不阻断 V1.0，但必须在文档中明确。

## 9. 阶段 7：正式执行版工作推进表

1. 阶段名称：正式执行版工作推进表。
2. 通俗理解：项目 active 后，Task / SubTask 成为执行管理的主数据。
3. 主要角色：owner、coordinator、member、super_admin / tech_admin。
4. 入口页面：`/work/tasks`，前端 `TaskManagementPage`，含执行视图与计划表视图。
5. 核心操作：查看重点工作、关键任务、计划表、进展记录。
6. 核心产物：正式执行版工作推进表。
7. 数据对象：Task、SubTask、UpdateSubmission、OperationLog。
8. 状态变化：任务状态如未开始 / 进行中 / 已完成 / 延期 / 暂缓等。
9. 下一阶段入口：关键任务派发、成员提交进展。
10. 该阶段允许做什么：执行期查看、补派/调整关键任务、查看回流进展。
11. 该阶段禁止做什么：承担立项审批职责；替代 AI 确认入库。
12. 当前代码是否已经支持：支持。`TaskManagementPage` 已有执行视图 / 计划表视图。
13. 当前代码差异 / 风险：后端 SubTask 结构写入主要校验 owner/coordinator/tech_admin 和非 archived，active 限制主要由前端 `isProjectActive` 执行；后续需审计是否需要后端也强制 active。

## 10. 阶段 8：关键任务派发

1. 阶段名称：关键任务派发。
2. 通俗理解：active 项目中，负责人 / 统筹人可以补派关键任务给具体责任人。
3. 主要角色：owner、coordinator、tech_admin；member 不能结构性派发。
4. 入口页面：TaskManagementPage 的“新增关键任务 / 编辑关键任务”。
5. 核心操作：为某个重点工作新增或编辑 SubTask，指定责任人、协助人、时间段、备注 / 标准。
6. 核心产物：SubTask。
7. 数据对象：Task、SubTask、ProjectMember。
8. 状态变化：新增有责任人的关键任务可进入进行中；父任务状态会同步。
9. 下一阶段入口：成员在工作汇报页按项目/关键任务提交进展。
10. 该阶段允许做什么：active 项目执行期补派关键任务。
11. 该阶段禁止做什么：普通 member 派发结构性任务；为 archived 项目新增任务；新增第四层任务。
12. 当前代码是否已经支持：前端支持，后端支持 owner/coordinator/tech_admin 创建/编辑 SubTask。
13. 当前代码差异 / 风险：后端未显式要求 project active，只拦截 archived；这是 P1 差异，建议后续审计是否补后端 active 校验。

## 11. 阶段 9：成员提交工作进展

1. 阶段名称：成员提交工作进展。
2. 通俗理解：成员围绕明确项目，理想情况下围绕明确关键任务，提交本周完成/成果/问题/下周计划。
3. 主要角色：member、关键任务 assignee、owner 也可提交。
4. 入口页面：`/work/submit`，前端 `VoiceUpdatePage`。
5. 核心操作：选择所属项目、输入文本、调用 AI 提取、提交给负责人。
6. 核心产物：UpdateSubmission 草稿 / 待确认记录。
7. 数据对象：UpdateSubmission、Project、SubTask 上下文。
8. 状态变化：生成 `confirm_status=待确认`。
9. 下一阶段入口：AI 提取结果进入确认中心。
10. 该阶段允许做什么：成员提交进展；AI 辅助提取内容。
11. 该阶段禁止做什么：没有项目归属时提交正式汇报；非 active 项目提交正式汇报；由 AI 决定项目归属作为主路径。
12. 当前代码是否已经支持：项目绑定和 active 拦截已支持。前端 `VoiceUpdatePage` 选择项目，后端 `_require_project_active()` 拦截非 active。
13. 当前代码差异 / 风险：后端仍保留未传 `project_id` 时从 AI / task_reports 反查项目的兼容路径；理想流程中这只能是兼容兜底，不能成为主路径。关键任务绑定目前还不是正式提交的强制前置条件，属于 P0/P1 边界风险。

## 12. 阶段 10：AI 提取

1. 阶段名称：AI 提取。
2. 通俗理解：AI 把成员原始文字提取成结构化草稿。
3. 主要角色：member 发起，AI 提取，owner 后续审核。
4. 入口页面：VoiceUpdatePage。
5. 核心操作：调用 `/api/updates/extract`，生成 suggestion。
6. 核心产物：AI 提取草稿，包含进展、成果、问题、任务卡片等。
7. 数据对象：ExtractRequest、UpdateSubmission.ai_result_json。
8. 状态变化：提取本身不产生正式业务状态。
9. 下一阶段入口：提交后进入 AI 确认中心。
10. 该阶段允许做什么：提取内容、辅助识别关键任务。
11. 该阶段禁止做什么：决定项目归属；直接写入成果库/问题中心；替代负责人确认。
12. 当前代码是否已经支持：支持。
13. 当前代码差异 / 风险：提取请求可带 `project_id`；但后端提取接口只做项目访问校验，正式 active 拦截在创建提交时执行。需继续保持“提取不是入库”的边界。

## 13. 阶段 11：负责人确认入库

1. 阶段名称：负责人确认入库。
2. 通俗理解：负责人查看 AI 提取草稿，决定确认、退回、转交统筹人、上报企业教练。
3. 主要角色：owner；coordinator 处理统筹反馈；project_ceo 处理项目企业教练决策；super_admin / tech_admin 兜底。
4. 入口页面：`/work/confirmations`，前端 `ConfirmPage`。
5. 核心操作：确认入库、退回修改、转交统筹人、上报企业教练。
6. 核心产物：正式写入任务进展 / 成果 / 问题，或形成退回/转交状态。
7. 数据对象：UpdateSubmission、Task、SubTask、Achievement、Issue。
8. 状态变化：`待确认` → `已入库` / `已打回提交人` / `已转交统筹人` / `待CEO决策` 等。
9. 下一阶段入口：成果库、问题中心、工作推进表、驾驶舱查看回流结果。
10. 该阶段允许做什么：审核 AI 草稿并写入正式数据。
11. 该阶段禁止做什么：普通 member 确认别人的提交；把“我的提交”当成确认中心主职责。
12. 当前代码是否已经支持：支持。后端 `/api/confirmations/pending` 按当前登录用户权限过滤；前端已有“我的提交 / 待确认”双视图。
13. 当前代码差异 / 风险：确认中心仍保留“我的提交”视图，应定位为辅助记录；主职责必须是 reviewer 的“待确认处理台”。

## 14. 阶段 12：成果 / 问题 / 工作推进表 / 驾驶舱回流

1. 阶段名称：确认后数据回流。
2. 通俗理解：负责人确认后，结构化结果才成为正式数据，并被各业务页面展示。
3. 主要角色：owner、member、coordinator、project_ceo、管理层。
4. 入口页面：`/work/achievements`、`/work/issues`、`/work/tasks`、`/home/dashboard`。
5. 核心操作：查看成果、问题、任务进展、统计。
6. 核心产物：正式成果记录、正式问题记录、任务/关键任务进展、驾驶舱统计。
7. 数据对象：Achievement、Issue、Task、SubTask、UpdateSubmission。
8. 状态变化：UpdateSubmission 已入库；成果/问题进入各自状态流。
9. 下一阶段入口：执行持续推进或项目结束归档。
10. 该阶段允许做什么：查询、跟进、统计。
11. 该阶段禁止做什么：展示未确认 AI 草稿作为正式成果/问题；让成果库或问题中心承担 AI 草稿审核职责。
12. 当前代码是否已经支持：部分支持。确认逻辑会写入 Achievement / Issue / Task / SubTask，页面已有对应列表。
13. 当前代码差异 / 风险：仍需单独验收“确认入库后所有模块回流完整性”，尤其是驾驶舱统计和工作推进表记录展示是否与确认结果一致。

## 15. 阶段 13：项目结束与归档

1. 阶段名称：项目结束与归档。
2. 通俗理解：项目完成后归档，转为只读历史资料。
3. 主要角色：super_admin / tech_admin / 有归档权限的管理者。
4. 入口页面：ProjectsMgmtSection。
5. 核心操作：归档、必要时恢复。
6. 核心产物：archived 项目档案。
7. 数据对象：Project。
8. 状态变化：任意非 archived 状态 → `archived`；恢复后需按规则回到合适状态。
9. 下一阶段入口：归档档案查看。
10. 该阶段允许做什么：查看历史、恢复（如有权限）。
11. 该阶段禁止做什么：继续正式汇报；继续派发任务；继续写入成果/问题。
12. 当前代码是否已经支持：部分支持。`archive_project()` 设置 archived；多处 `require_project_not_archived()` 拦截写入。
13. 当前代码差异 / 风险：需要继续审计所有执行期写入口是否都拦截 archived；当前 updates、achievements、issues、subtasks 等已有部分保护。

## 17. P0 / P1 / P2 差异清单

### P0：阻断主流程正式使用的问题

1. 成员正式提交工作进展是否必须绑定关键任务尚未最终收口。  
   - 业务基准：理想状态下应有明确项目归属和关键任务归属。
   - 当前事实：项目归属已被强化；关键任务归属仍主要作为上下文/选择项。
   - 影响：成员可能在 active 项目中提交泛化汇报，负责人确认后难以准确回流到工作推进表。

### P1：不阻断但影响正式使用的问题

1. 缺少独立启动确认 / 启动会状态。
2. AI 确认中心的“我的提交”视图容易稀释“待确认处理台”职责。
3. 执行期补派关键任务后端 active 边界不够硬。
4. 确认入库后回流到成果 / 问题 / 工作推进表 / 驾驶舱的完整性仍需专项验收。
5. 旧导入/种子路径可能绕过 draft 默认。
6. archived 项目的所有写入口是否完整拦截仍需全量复查。

### P2：体验优化或后续增强

1. 启动会纪要和项目生命周期状态的关联展示。
2. 工作推进表执行版的版本化或变更记录。
3. 我的提交记录应迁移到工作汇报 / 我的任务更合适的位置。
4. 项目归档档案的完整只读视图。

## 19. 模块职责边界

### 19.1 项目管理

负责：

- 创建项目。
- 配置角色。
- 下发项目。
- 审核立项。
- 生命周期流转。

不负责：

- 执行期日常进展。
- 成员提交进展。
- 成果/问题入库。

### 19.2 立项填写工作台 / OwnerSubmitModal

负责：

- 负责人完善项目完成准则。
- 编制工作推进表雏形。
- 提交企业教练审核。

不负责：

- 成员日常汇报。
- 确认入库。
- 执行期任务管理。

### 19.3 工作推进表

负责：

- 展示正式执行版工作推进表。
- 展示重点工作 / 关键任务。
- 执行期补派 / 调整关键任务。
- 查看进展回流。

不负责：

- 立项生命周期审批。
- AI 确认入库。

### 19.4 工作汇报

负责：

- 成员提交进展。
- 绑定项目。
- 理想状态下绑定关键任务。
- 调用 AI 提取。

不负责：

- 最终入库。
- 成果库 / 问题中心直接写入。

### 19.5 AI 确认中心

负责：

- 负责人 / 统筹人 / 企业教练处理待确认 AI 提取结果。
- 确认入库。
- 退回修改。
- 转交统筹人或上报企业教练。

不负责：

- 成员查看“我的提交”主列表。
- 创建工作汇报。

### 19.6 成果库

负责：

- 查看确认后的成果。
- 支持成果后续管理。

不负责：

- AI 草稿展示。
- 未确认成果正式入库。

### 19.7 问题中心

负责：

- 查看确认后的问题。
- 后续问题跟进。

不负责：

- 未确认草稿问题正式入库。

### 19.8 驾驶舱

负责：

- 汇总状态。
- 展示统计。
- 管理层查看。

不负责：

- 具体录入。
- 确认入库。

## 20. 关键业务口径

1. 项目成员不等于关键任务责任人。
2. 项目 active 不等于每个成员都有可汇报任务。
3. 成员正式提交工作进展前，应该有明确项目归属。
4. 更理想情况下，成员提交工作进展前还应该有明确关键任务归属。
5. AI 不应该作为项目归属判断的主路径。
6. AI 负责提取内容，不负责决定业务归属。
7. AI 提取结果不是正式数据。
8. 负责人确认后才写入正式数据。
9. AI 确认中心应该是“待确认处理台”，不是“我的提交记录中心”。
10. 我的提交记录应归属工作汇报 / 我的任务，而不是 AI 确认中心主职责。
11. 工作推进表雏形不是最终执行版。
12. 启动会的核心材料是工作推进表雏形。
13. 启动会的本质是把雏形讨论成正式执行版。
14. 如果 V1.0 暂不做强制启动会状态，则必须说明：企业教练审核通过 = 当前工作推进表雏形确认为执行版，项目进入 active。
15. 后续正式增强版可以增加：待启动 / 启动确认 状态。

## 21. 明确禁止事项

后续修改禁止：

1. 新增第四层任务。
2. 新增 Workstream 表。
3. 把 Task / SubTask 语义改乱。
4. 让 AI 成为项目归属主路径。
5. 让成员在没有项目归属时提交正式汇报。
6. 让非 active 项目提交正式汇报。
7. 把 AI 确认中心做成我的提交记录中心。
8. 把项目管理页做成工作推进表。
9. 把工作推进表做成立项审批页。
10. 继续“发现一个页面问题就修一个页面问题”，而不先检查主流程阶段边界。
