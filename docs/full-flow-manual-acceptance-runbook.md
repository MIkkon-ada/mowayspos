# 全流程人工验收 Runbook

> N4-P0-D-FULL-FLOW-MANUAL-ACCEPTANCE-RUNBOOK

## 1. 验收目标

本次验收**不是开发新功能**，而是从用户视角人工验证 Moways 项目协同平台的完整内部项目闭环：

```
创建项目 → 下发PM → 补立项信息 → 工作推进表雏形 → 提交审核 → 企业教练审核通过 → active
    → 工作推进表（执行视图/计划表视图）→ 成员提交进展 → AI确认中心 → owner确认入库
    → 成果库 / 问题中心 → 工作推进表复查 → 权限负向验证
```

目标：确认系统已支持一条完整的内部项目从启动到执行再到成果沉淀的闭环，无 P0 阻断性缺陷。

---

## 2. 验收前准备

### 2.1 本地启动方式

**方式一：一键启动（推荐）**

在项目根目录执行：

```
start-dev.bat
```

这会同时打开两个终端窗口：
- 后端：`http://127.0.0.1:8008`
- 前端：`http://127.0.0.1:6001`

等待约 6 秒后自动打开浏览器。

**方式二：分别启动**

```bash
# 终端1 - 后端
cd bowei_ai_dashboard
call .venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8008

# 终端2 - 前端
cd frontend
npm run dev
```

### 2.2 服务端口

| 服务 | 端口 | 说明 |
|------|------|------|
| 后端 API | `8008` | FastAPI + uvicorn |
| 前端 | `6001` | Vite dev server |

### 2.3 数据库

- 使用本地 PostgreSQL（Docker Compose 管理）
- `docker-compose.yml` 需已启动
- **不需要**为本轮验收清理旧数据
- **建议**使用一组新的测试项目名称，避免与既有项目混淆

### 2.4 登录账号

- 系统使用 API Key 登录
- 不同角色需使用不同的 API Key
- **不要提交 `.env` 文件或真实密码到仓库**

### 2.5 不需要

- 不需要真实 AI 服务（规则引擎兜底即可）
- 不需要真实客户数据
- 不需要部署到生产环境
- 不需要修改 `.env` 配置（除非特定配置缺失导致无法启动）

## 3. 建议测试角色

| 序号 | 角色 | 系统中对应 | 验收用途 |
|------|------|-----------|---------|
| R1 | super_admin / tech_admin | `is_tech_admin = true` | 创建项目、配置人员、技术兜底启动 |
| R2 | company_ceo（纯公司 CEO） | `is_ceo = true`，不在 `project_members` | 验证**不能**越权当 project_ceo/owner |
| R3 | project_ceo（企业教练） | `project_members.role = "project_ceo"` | 审核立项（通过/退回） |
| R4 | owner（负责人/PM） | `project_members.role = "owner"` | 补立项、确认入库、查看工作推进表 |
| R5 | coordinator（统筹人） | `project_members.role = "coordinator"` | 验证**不能**替代 owner 做最终确认 |
| R6 | member（成员） | `project_members.role = "member"` | 提交进展、验证不能越权确认 |

**关键约束：**

- `company_ceo` ≠ `project_ceo`。只有 `project_members` 中角色为 `project_ceo` 的用户才是企业教练。
- `project_ceo` 必须来自项目成员表（`project_members.role`），而非单纯 `is_ceo` 标志位。
- 验收时需要**至少 3 个独立用户账号**（最小配置：super_admin + owner + member），理想情况 4-5 个。

## 5. 全流程验收步骤

### A. 登录

**步骤：**

1. 打开浏览器访问 `http://127.0.0.1:6001`
2. 使用 **super_admin** API Key 登录
3. 登录成功后进入首页，标题为"博维 AI 升级项目驾驶舱"（案例项目名）

**预期：**

### B. 创建项目

**路径：** 导航至 `系统设置` → `项目管理` 或直接访问 `/home/projects`

**步骤：**

1. 使用 **R1（super_admin）** 登录
2. 点击「新建项目」按钮
3. 填写项目基本信息：
   - 项目名称：`AI升级转型验收项目`
   - 项目类型：选择"博维内部项目"
   - 项目背景：填入 4.2 节背景内容
   - 项目目标：填入 4.3 节目标内容
   - 预期交付物：填入 4.4 节交付物
   - 计划开始：`2026-07-01`
   - 计划结束：`2026-12-31`
4. **配置项目角色：**
   - 企业教练（project_ceo）：选 **R3**
   - 负责人（owner）：选 **R4**
   - 统筹人（coordinator）：选 **R5**（可选）
   - 成员（member）：选 **R6**
5. 保存，项目状态为 **draft**（草稿）

**预期：**
- 项目出现在「草稿」Tab 下
- 项目卡片显示：名称、类型、状态 badge
- 操作按钮：下发项目、编辑项目
- （此时 project_ceo 和 owner 均无法看到此项目，因为项目尚未下发）

---

### C. 下发给 PM

**前置：** super_admin 登录，项目处于 draft 状态

**步骤：**

1. 在项目管理页面，找到"AI升级转型验收项目"
2. 确认项目已配置企业教练（project_ceo）和负责人（owner）
3. 点击「下发项目」

**预期：**
- 项目状态从 `draft` 变为 `dispatched`
- 项目出现在「已派发」Tab 下
- **R3（project_ceo）** 和 **R4（owner）** 现在可以看到此项目
- R4（owner）的项目卡片上出现「完善立项信息」按钮

### D. PM 补立项信息

**前置：** 项目 dispatched，使用 **R4（owner）** 登录

**路径：** `/home/projects`

**步骤：**

1. 使用 **R4（owner）** API Key 登录
2. 导航至项目管理页面
3. 找到"AI升级转型验收项目"
4. 点击「完善立项信息」
5. 在弹窗中填写/确认：
   - 项目类型：博维内部项目（已预填，可改）
   - 项目背景：确认或补充
   - 项目目标：确认或补充
   - 预期交付物：确认或补充
   - 计划时间：确认或修正
6. **填写工作推进表雏形**（见步骤 E）
7. 点击「提交立项审核」

**预期：**
- 项目状态变为 `pending_review`
- 项目出现在「待审核」Tab 下
- 弹窗关闭后 toast 提示："已提交审核，等待企业教练审核通过后正式启动"

---

### E. 工作推进表雏形

**前置：** 在 OwnerSubmitModal 弹窗中（步骤 D）

**数据结构说明：**
- **重点工作**（Task）：项目层面的主要工作方向，用于归类。仅需填写标题和说明/评价标准。
- **关键任务**（SubTask）：重点工作下的具体可执行任务。需要明确责任人、协助人、计划时间。

**步骤：**

1. 在弹窗中，点击「新增重点工作」
2. 填入第 4.6 节的 3 个重点工作：
   - 条目 1：开发并应用项目运营系统
   - 条目 2：建立 AI 成果与问题沉淀机制
   - 条目 3：形成内部项目推进标准流程
3. 每个重点工作只需填写：
   - **重点工作标题**
   - **重点工作说明 / 评价标准**
4. 在每个重点工作下，点击「新增关键任务」
5. 填入各关键任务的：
   - 关键任务标题
   - 评价标准
   - **责任人**（关键任务层保留）
   - **协助人**（关键任务层保留，如有）
   - **计划开始时间**（关键任务层保留）
   - **计划结束时间**（关键任务层保留）

**验收点：**
- 至少 1 个重点工作（推荐 3 个）
- 至少 1 个关键任务（推荐每项 ≥ 2 个）
- 重点工作层**不显示**责任人和协助人输入框（仅标题和说明）
- 关键任务层**保留**责任人、协助人、计划时间等可执行字段
- 说明文案明确："重点工作用于归类工作方向；关键任务才需要明确责任人、协助人和计划时间。"
- 表单提交前会过滤空标题行
- 提交时校验：至少 1 条重点工作 + 至少 1 条关键任务

### F. PM 提交审核

3. PM 不能看到「审核通过」按钮（该按钮仅 project_ceo/TechAdmin 可见）

### G. 企业教练审核

**前置：** 项目 pending_review，使用 **R3（project_ceo）** 登录

**路径：** `/home/projects`

**步骤：**

1. 使用 **R3（project_ceo）** API Key 登录
2. 导航至项目管理页面

4. 点击「审核立项」
5. 在审核弹窗中查看：
   - 项目背景、目标、交付物
   - 工作推进表雏形（重点工作 + 关键任务列表）

7. 点击「审核并确立」（即审核通过）

**预期：**
- 项目状态变为 `active`
- 项目出现在「进行中」Tab 下
- 项目卡片显示 `kickoff_date` 为审核通过时间

**负向验证（企业教练退回）：**

1. 企业教练也可以在审核弹窗中点击「退回修改」
2. 输入退回原因
3. 项目状态变为 `returned`
4. PM 可以看到项目回到「已退回」Tab，可再次修改并提交

## 6. 权限负向验收

以下场景必须在验收中**逐一验证**，验证方法是使用相应角色登录并尝试操作。

| 序号 | 场景 | 操作角色 | 目标操作 | 预期结果 |
|------|------|---------|---------|---------|
| N1 | 普通成员不能审核立项 | member | 对 pending_review 项目点审核 | 无审核按钮，或被拒绝 |
| N2 | 普通成员不能确认入库 | member | 在确认中心对他人提交做确认 | 无确认按钮，或被拒绝 |
| N3 | 统筹人不能替代 owner 做最终确认 | coordinator | 在确认中心做确认入库 | 无确认按钮，或被拒绝（取决于实际权限配置） |

| N5 | company_ceo 不能越权确认入库 | company_ceo（非项目成员） | 在确认中心确认 | 被拒绝 |
| N6 | 非 active 项目不能提交正式汇报 | member | 对 draft/dispatched/pending_review 项目提交汇报 | 被拒绝（409 或 403） |
| N7 | 未确认提交不能直接进入成果库 | — | 提交后立即查看成果库 | 成果库中无此记录 |
| N8 | 未确认提交不能直接进入问题中心 | — | 提交后立即查看问题中心 | 问题中心中无此记录 |
| N9 | 未确认提交不能出现在工作推进表正式更新 | — | 提交后立即查看工作推进表 | 无此更新记录 |

```
新建项目 → draft（草稿）
              │
              └─ dispatch（下发）→ dispatched（已派发）
                                      │
                                      └─ owner-submit（PM提交）→ pending_review（待审核）
                                                                      │
                                                          ┌───────────┘
                                                          │
                                              ┌─ approve（审核通过）→ active（进行中）
                                              │                           │
                                              │                           └─ archive → archived（已归档）
                                              │
                                              └─ return（退回）→ returned（已退回）
                                                                      │
                                                                      └─ owner 修改后重新提交 → pending_review
```
