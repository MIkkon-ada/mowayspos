# P7-P2 跨项目工作汇报正式提交设计

## 目标

把 P7-P1 已校对且完成归属的多项目任务卡，通过一次原子请求正式提交。系统按项目生成现有 `UpdateSubmission` 子提交，继续复用现有负责人、统筹人、企业教练审核和入库流程。

## 范围

本阶段包含：

- 跨项目批次与子提交关联。
- 服务端按项目分组并原子创建子提交。
- 防重复提交。
- 工作汇报提交成功态和历史提交批次展示。
- 各项目子提交继续进入现有审核队列。

本阶段不改变：

- `POST /api/updates` 及前端 `createUpdate` 的单任务兼容路径。
- 审核状态常量和状态流转。
- 负责人、统筹人、企业教练权限语义。
- 单卡确认、退回、转交、反馈、批示和入库 handler。
- Agent 提取与匹配规则。

## 方案选择

采用服务端批次事务方案。一次请求创建一个批次，任务卡按项目分组；同项目多卡合成一条 `UpdateSubmission`，不同项目生成不同子提交。首次创建时任意校验或写入失败均整体回滚，创建成功后的各项目审核彼此独立。

不采用单条跨项目提交，因为 `UpdateSubmission.project_id`、审核权限和通知均以单项目为边界。不采用前端循环调用 `createUpdate`，因为它无法保证原子性、幂等性和整批追踪。

## 数据模型

新增 `update_submission_batches`：

- `id`：主键。
- `client_request_id`：客户端幂等标识，唯一索引。
- `submitter`、`submitter_id`：提交人快照与人员关联。
- `source_type`：来源类型。
- `title`：整次汇报标题。
- `transcript_text`：完整原文。
- `submission_count`：成功创建的项目子提交数量。
- `created_at`、`updated_at`：时间戳。

扩展 `update_submissions`：

- `batch_id`：可空外键，旧提交保持为空。
- `batch_order`：批次内稳定排序，默认 0。

批次不持久化新的审核状态。历史展示状态根据子提交的现有状态实时聚合，避免产生第二套状态机。

## 批量提交接口

新增 `POST /api/updates/batch`，不修改现有正式提交接口。

请求包含：

- `client_request_id`
- `source_type`
- `title`
- `transcript_text`
- `human_result`，其中 `task_reports` 为用户校对后的完整卡片集合

服务端只接受归属完整的卡片。每张卡必须包含可解析的 `parent_task_id` 和 `matched_subtask_id`。项目身份由数据库中的重点工作和关键任务反查，不信任前端项目名称。

响应包含批次摘要和按项目创建的子提交列表，供前端定位历史记录。

## 服务端处理流程

1. 验证登录人及提交人身份。
2. 通过 `client_request_id` 查找既有批次；存在时直接返回既有结果。
3. 校验任务卡列表非空且所有卡片归属已确认。
4. 对每张卡反查重点工作、关键任务和所属项目。
5. 校验重点工作属于项目、关键任务属于重点工作。
6. 校验当前用户对每个项目有提交权限，且项目未归档并处于可提交生命周期。
7. 按项目 ID 分组，保持卡片原始顺序。
8. 创建批次。
9. 每个项目创建一条 `UpdateSubmission`，其 `human_result.task_reports` 只包含该项目卡片；完整原文保留在每条子提交中。
10. 每个项目负责人生成一次待审核通知。
11. 单次提交事务；全部写入成功后统一提交。
12. 返回批次和子提交。

## 审核行为

审核中心仍只读取和处理 `UpdateSubmission`。每条子提交有唯一 `project_id`，因此现有项目负责人、统筹人和企业教练权限判断继续有效。

同项目多张卡继续使用现有单卡 handler。某项目退回只更新该项目子提交；其他项目保持原状态并继续审核。批次不会反向驱动或覆盖子提交状态。

## 历史提交

本人历史记录中：

- `batch_id` 为空的旧提交按现有单条记录展示。
- 同一 `batch_id` 的记录聚合为一次跨项目汇报。
- 主记录展示原文摘要、提交时间、来源、项目数量和聚合状态。
- 展开后展示每个项目名称、任务卡数量、当前审核状态和子提交详情入口。

展示聚合规则：

- 所有子提交已入库：已确认。
- 任一子提交处于退回状态：已退回或部分退回，并展示各项目明细。
- 其余存在审核流转中的子提交：审核中。
- 撤回、不入库等终态只在批次明细中按现有状态标签展示，不新增状态常量。

## 前端提交路径

- `task` 范围继续调用 `createUpdate`。
- `all` 和 `project` 范围在所有卡片归属完整后调用新的批量提交接口。
- 前端生成并在一次提交尝试中复用 `client_request_id`，防止重复点击或网络重试产生重复批次。
- 提交过程中锁定操作；成功后进入现有完成阶段并打开历史提交定位到新批次。
- 任一归属未完成时按钮保持禁用，不向后端发送请求。

## 错误处理

- 校验错误返回卡片索引和明确原因。
- 任一项目无权限、已归档、生命周期无效或归属不一致时整体回滚。
- 数据库唯一约束处理并发的重复 `client_request_id`。
- 通知与批次、子提交处于同一事务中，失败不遗留部分业务记录。
- 前端失败后保留原文、提取结果、卡片切换位置和人工修改内容。

## 测试与验收

后端测试：

- 两项目生成两个子提交。
- 同项目多卡只生成一个子提交。
- 子提交只包含本项目任务卡。
- 任一卡片归属无效时零写入。
- 任一项目无权限或已归档时零写入。
- 相同 `client_request_id` 不重复创建或通知。
- 项目负责人只能审核对应项目子提交。
- 一个项目退回不改变其他项目子提交。
- 旧 `create_update` 测试继续通过。

前端测试：

- 全部工作和指定项目范围调用批量接口。
- 指定关键任务继续调用 `createUpdate`。
- 未确认归属不能提交。
- 请求按当前校对结果发送全部卡片。
- 重复点击只发送一次。
- 失败保留草稿。
- 成功后历史提交定位批次。
- 历史记录按批次聚合并展示各项目状态。

验收执行后端 compileall、相关 pytest、前端 build、全部 `.mjs` 测试、数据库迁移升级/降级验证、真实浏览器跨项目提交及两个项目审核队列检查。截图保存在仓库外。

## 安全与兼容

- 不信任前端提交的项目名称、角色或状态。
- 不允许提交其他用户无权汇报的任务。
- 不修改现有状态常量。
- 不修改现有审核 handler 的业务含义。
- 旧数据、旧路由、单任务深链和单任务提交保持兼容。
- 不 commit、不 push、不创建 PR。
